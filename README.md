# Dungeon Event Processor

Система обработки логов событий в подземелье. Программа считывает конфигурацию подземелья и поток событий, после чего выводит лог обработки и финальный отчет по каждому игроку.

## Технологический стек

- **Язык**: Go 1.26.1
- **Библиотеки**:
  - `github.com/ilyakaznacheev/cleanenv` — для удобного чтения конфигурации из JSON.
- **Инструменты**: Makefile для автоматизации сборки и тестирования.

## Архитектура

Проект разбит на логические слои:

- `cmd/dangeon`: точка входа, обработка флагов и чтение входного потока.
- `internal/config`: загрузка и расчет параметров подземелья (время открытия/закрытия).
- `internal/domain`: описание базовых сущностей (Player, Event, Floor) и логика формирования отчета.
- `internal/event_parser`: парсинг строк лога с валидацией данных и поддержкой перехода через полночь.
- `internal/processor`: «мозг» системы, реализующий правила подземелья и переходы состояний игроков.

## Как запустить

### С помощью Makefile (рекомендуется)

Для сборки и запуска с параметрами по умолчанию:

```bash
make run
```

Вы можете передать свои пути к файлам через переменные:

```bash
make run CONFIG_PATH=./my_config.json EVENTS_PATH=./my_events.txt
```

Для запуска тестов:

```bash
make test
```

### С помощью Go Run

```bash
go run cmd/dangeon/main.go --config config/config.json --events events
```

- `--config`: путь к JSON файлу конфигурации (обязательно).
- `--events`: путь к файлу с логами (опционально, если не указан — чтение идет из `stdin`).

## Тонкости реализации

### 1. Завершение логов (Edge Case)

Одной из ключевых задач была обработка ситуации, когда лог событий заканчивается, но игроки еще не вышли из подземелья.

- **Решение**: В системе реализован метод `CloseDungeon()`, который вызывается после обработки всех доступных событий.
- Если время подземелья истекло или логи закончились, все активные игроки автоматически получают статус `FAIL` (если они вошли в подземелье) или `DISQUAL` (если даже не начали путь).
- Время их пребывания в этом случае рассчитывается до момента планового закрытия подземелья (`OpenAt + Duration`).

### 2. Учет времени на этажах

Согласно правилу время, проведенное на уже зачищенном этаже, не учитывается.

- В реализации `TotalTimeSpent` для этажа фиксируется только в момент его завершения (убийство всех монстров или босса). Любые последующие заходы на этот этаж (например, при возвращении назад) не увеличивают счетчик времени для этого этажа.

### 3. Переход через полночь

Парсер событий умеет определять ситуацию, когда время в логах переходит через отметку `00:00:00` (следующие сутки), и корректно увеличивает смещение времени (`daysOffset`), чтобы расчеты длительности оставались точными.
